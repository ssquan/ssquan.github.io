Presto Insert Mysql

September 16, 2020

故事是这样开始的

一个咖啡劲儿已经消退了的午后，业务大大突然抛了一个非常令人吃惊的问题，说: Presto insert Mysql 特别慢，跑了几十分钟了，一直 13% 左右， 你赶紧帮忙看看。第一反应就是 嗯？大问题了？，有一点兴奋，也有一点冒汗。

一顿操作猛如虎

紧接着，大概进行了如下几个猛如虎的操作：

  1. Presto 监控看起来，嗯～有一个 Stage 很慢，但是又没有死，大概几十 K Row/s 的速度在蠕动；
  2. Mysql 监控看起来，哦？CPU 低很，内存不大，磁盘占用率和 IO 都不高，网络更别说，不忙；
  3. Presto 进程在搞啥这么慢？Jstack 一下，stack 就是在 write 的地方；
  4. Mysql 的信息稍微捞一捞。

果然不出所料，没那么 easy 就看到原因，再来一杯咖啡提提神吧。

来自大自然的猜想

在经过上面的操作之后，基本确定的是两个系统都没有发生什么异常，他们在正常的工作，只是慢。正常操作，资源不紧张，看起来不是瓶颈，深度怀疑是有竞争了。难道是有锁竞争？Show Create Table 还是有一点小小的惊喜，内容是这样的（ 已脱敏）：

动手试验

Presto Insert Mysql 的流程

Presto 通过先将数据写入临时表的方式达到了 atom 的效果，具体流程如下：

sequenceDiagram
    participant presto_coordinator
    participant presto_worker
    participant Mysql
    
    presto_coordinator ->> Mysql: 1. Create temporary table
    presto_coordinator ->> presto_worker: 2. Write temp table
    presto_worker ->> Mysql: 3. Insert into temp table
    presto_coordinator ->> Mysql: 4.Insert into as select from temporary table
    presto_coordinator ->> Mysql: 5. Delete temporary table

步骤文字说明：

  1. 在 Presto Coordinator 解析作业时如果发现是 Insert 操作，则创建一个临时表；
  2. Presto Worker 上开始运行作业，读取数据，写入数据；
  3. Presto Coordinator 在数据写完后，执行 SQL 将数据从临时表拷贝到目标表；
  4. Presto Coordinator 删除临时表；

临时表的做法带来了原子性，也伴随着两个问题：

  1. 写放大，整个过程，Mysql 需要至少 2 倍于数据量的空间；
  2. 临时表有些情况会被残留，必现的场景是执行操作过程中 Mysql 发生重启；
